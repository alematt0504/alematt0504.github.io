LLMs EVALUATION IN PREDICTING JAVA CODE CONFIGURATIONS

This project investigates whether a large language model can correctly infer and generate a Checkstyle configuration file simply by observing the stylistic patterns present in Java source code. The central idea is to determine whether an artificial model, without direct access to a project’s original Checkstyle rules, can reconstruct an equivalent configuration that governs indentation, naming conventions, structural organization, and other stylistic constraints embedded in real repositories. Since Checkstyle is widely used in collaborative software projects to enforce coding standards, automating its configuration represents a compelling challenge in both information retrieval and code-oriented personalization. The study therefore aims to measure how reliably an LLM recognizes latent style patterns and whether it can formalize them into a syntactically correct and semantically meaningful XML structure.

Methodology:

To address this objective, the work begins with the construction of a large dataset of GitHub repositories that actively employ Checkstyle. This requires overcoming several technical obstacles, such as API request limits, incomplete or low-quality repositories, and the difficulty of associating detected configuration files with their original projects. Through iterative refinement, a final dataset of hundreds of valid repositories is assembled, each containing a consistent Checkstyle.xml and a set of Java files conforming to it. Once curated, this dataset becomes the foundation for extracting both the ground-truth configurations and the source files that will serve as input to the LLM. An important methodological refinement is the exclusion of files suppressed intentionally by configuration filters, ensuring that the model only analyzes Java files that truly reflect the project’s style rules. After retrieving and preparing the Java code, the next stage consists of using an LLM to generate predicted Checkstyle configurations. Prompt engineering plays a critical role here, as the model must be guided to respect the hierarchical structure of Checkstyle, separating global modules from TreeWalker modules and maintaining the correct XML organization. The prompt is progressively improved to reduce structural errors and ensure more consistent predictions. Additional constraints, such as deterministic decoding and token-limit handling, help stabilize output quality and reproducibility. With both the true and predicted Checkstyle configurations available, the evaluation framework compares them through a dual strategy. First, a structural comparison assesses how accurately the predicted modules match the true ones, accounting for false positives, false negatives, and true positives. A tailored loss function is developed to reflect the varying difficulty of inferring different modules and to normalize results across projects of diverse sizes. Second, an execution-based evaluation applies both configurations to Java files unseen by the LLM, measuring how many stylistic violations each configuration produces. By analyzing errors per file rather than raw error counts, the assessment becomes robust to differences in repository size.

Results:

The results suggest that LLMs can frequently produce syntactically valid Checkstyle files and often replicate project-specific style conventions with surprising accuracy. A substantial portion of generated configurations match the behavior of the true configuration on unseen code, while the rest tend to introduce stricter stylistic constraints rather than incorrect or inconsistent rules. These findings highlight both the promise and the current limitations of automated Checkstyle generation, pointing toward future improvements in prompt design, module inference, and fine-tuning strategies.
